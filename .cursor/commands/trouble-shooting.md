# トラブルシューティング — 状況の聞き取り・診断（Jarvis）

連携・同期・取り込み・送信などが **期待どおり動かない** ときの入口。  
呼び出し: **`/trouble-shooting`**（スラッシュコマンド）

方針の正本:

- `~/git-repos/.cursor/rules/jarvis-when-feature-fails-ask.mdc`
- `~/git-repos/.cursor/rules/jarvis-integration-fail-self-check.mdc`

---

## 第1ターン（必須）— 質問項目を並べて聞き取る

コマンドが呼ばれたら、**まず診断や修正に入らず**、次の **質問リスト** をそのまま見やすく出力する。  
前置きは1〜2文でよい（例:「次の項目に答えてもらえると、切り分けが早くなります。分かるところだけで大丈夫です。」）。

### 出力する質問（この形式で番号付き）

**1. 何ができるはずでしたか？（期待）**  
　例: 「815オプチャのスレッドも `5.やり取り.md` に追記されるはず」

**2. 実際にはどうなっていますか？（いま見えている症状）**  
　例: 「メインだけ増える」「追記0のまま」「エラーで止まる」

**3. どの機能・どこが対象ですか？**  
　例: スクリプト名、グループ名（30空室相談G）、`5.やり取り.md`、パートナー確認のオプチャ同期 など

**4. いつからそうなりましたか？**  
　例: 初めて / 昨日まで動いていた / 何度も同じ

**5. 補足（分かる範囲で）**  
　実行したコマンド、QRの有無、画面やターミナルに出たメッセージ（401、404 など）

---

**締めの一文（必須）:**  
「上の番号に沿って、分かるところから教えてください。音声入力のままでも大丈夫です。」

### 同じメッセージに状況が既に書いてある場合

- 上記 **質問リストは省略しない**（ユーザーが見ながら整理できるよう、必ず一度出す）
- その直後に **「すでに分かっていること」** を箇条書きで要約し、**足りない番号だけ** 追加で聞く（最大2項目まで）
- 5項目そろっていれば第2ターンへ進んでよい

---

## 第2ターン以降 — 整理・自律診断

ユーザーの回答を受け取ったら:

1. **依頼の整理版** — 5項目を1ブロックにまとめて提示（コピーしやすく）
2. **自律診断** — `jarvis-integration-fail-self-check.mdc` に従い Phase 0（dry-run・件数 grep・エラー分類・state/YAML）を **確認待ちで止まらず** 実行
3. **報告** — 仮説・実施した診断・次の一手（数値・エラーコードを含む）

不足があれば、追加質問は **最大2つまで**。

---

## Plan モード（方針が割れそうなとき）

Phase 0 報告の **直後**、またはユーザーが **方針で迷っている** とき、次の **Plan モード要否** を判断する（`jarvis-integration-fail-self-check.mdc` の「Plan モード判断」と同じ）。

### Jarvis が Plan がよさそうと判断したら（どちらか必須）

| 優先 | 動作 |
|------|------|
| **1（推奨）** | **`SwitchMode`（`target_mode_id: plan`）** を呼び、1文で理由を付けて Plan へ切り替えを **申請**する（Cursor の承認で Plan になる） |
| **2（フォールバック）** | `SwitchMode` が使えない、または **同セッションで Plan 切替を拒否済み** のときだけ、チャットで **「Plan モードに切り替えて、対応案を整理しましょうか？」** と提案する |

**いきなり大改修・大量上書きに入らない。** Plan では選択肢・トレードオフ・推奨案を出し、ユーザーが方針を選んでから Agent に戻って実装する。

### ユーザーが方針で迷ったとき（明示）

次のような発話があったら、上記 **1 → 2** の順で Plan を提案する（診断結果の A/B/C だけ並べて終わらない）。

- 「どれがいいかわからない」「迷っている」「方針を決めたい」
- 選択肢を見て **どれも選ばず** 様子見・質問のみ
- 「割り切りたい」「運用でカバーか実装か」

### Agent のまま進めてよいとき

- 次の一手が **1つにほぼ確定**（typo 修正、dry-run、件数 grep、QR 再実行 等）
- ユーザーが **「A を実行」「棚卸しから」** と既に指定した

---

## 制約

- 第1ターンでいきなり「直します」「できません」だけで終わらない
- 音声入力の誤変換は `jarvis-voice-input.mdc` に従い意図を優先
- 認証（QR）・大量上書き・正本の競合はユーザー確認が必要
