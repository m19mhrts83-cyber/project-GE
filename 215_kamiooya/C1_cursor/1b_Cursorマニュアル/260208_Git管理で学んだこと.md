# Git管理で学んだこと（2026/02/08）

Cursor で Git を使いながら覚えた用語と操作のメモ。

---

## 1. エクスプローラーに出るマークの意味

| マーク | 意味 |
|--------|------|
| **U** | 未追跡（Untracked）＝ まだ Git に追加していない新規ファイル |
| **丸（●）** | そのフォルダの中に「変更あり」または「未追跡」のファイルがある |

マークが出ている ＝ まだコミットしていない変更がある。区切りがよかったらコミットする。

---

## 2. コミットのタイミング

- **都度コミット**でも **その日の終わりにまとめてコミット** でもよい。
- まとめてコミットする場合、「何をした日だったか」がメッセージに残るので履歴がわかりやすい。

---

## 3. コミット前に変更内容を確認する

1. **Ctrl+Shift+G**（Mac は **Cmd+Shift+G**）でソース管理を開く
2. **「変更」** または **「ステージした変更」** のファイル名を **クリック**
3. 差分ビューが開く（左＝変更前、右＝変更後。緑＝追加、赤＝削除）
4. コミットするのは **「ステージした変更」** だけなので、コミット前にここを一通りクリックして確認するとよい

---

## 4. ステージとは

- **ステージ** ＝ 「この変更を、次のコミットに含める」と Git に伝えること
- ソース管理で、ファイル名の横の **「+」** を押すとステージされる
- **「ステージした変更」** の横の **「−」** でステージを解除できる（変更は残るが、今回のコミットには含めない）
- 全部まとめてコミットするときは、「変更」の横の **「+」**（すべてステージ）で一括ステージできる

---

## 5. コミットメッセージの提案（キラキラマーク）

- コミット欄にある **キラキラしたマーク** を押すと、**変更内容からコミットメッセージを自動提案**してくれる
- 提案が英語でも、そのまま使うか、コピーして日本語に直してから使ってもよい
- 「変更内容に沿ったメッセージ」にすると、あとから履歴を見たときにわかりやすい

---

## 6. プッシュ・プル・プルリクエスト

| 用語 | 意味 | 今の運用 |
|------|------|----------|
| **プッシュ** | ローカルのコミットをリモート（GitHub 等）に送る | リモートを設定していなければ不要 |
| **プル** | リモートからコミットを取り込む | 同上 |
| **プルリクエスト（PR）** | ブランチの変更をレビューしてから本流にマージする仕組み | 一人・ローカルのみなら必須ではない |

- **今の運用**（自分だけ・ローカルのみ・コミット前に差分で確認）なら、**プッシュ・プル・PR は必須ではない**
- **Replit や別のクラウドのコードと連携するとき**は、そこが「リモート」になるので、**プッシュ・プルで同期**するイメージ
  - プッシュ ＝ 自分の PC の変更をリモートに送る
  - プル ＝ リモートで直した内容を自分の PC に取り込む

- 215・300・500・DX互助会・ProgramCode には **「コミットおすすめ＋メッセージ例を出すが、コミットは実行しない」** ルール（`.cursor/rules/git-commit-habit.mdc`）を配置済み
- AI が編集したあと、区切りのよいタイミングで「コミットしておくのがおすすめです」とメッセージ例が出る。コミットは自分で行う。

## 9. コミットが止まったとき（300フォルダ・OneDrive タイムアウト）

**症状**  
300 フォルダ（OneDrive 上）で `git add` やコミットをすると、「Operation timed out」「failed to insert into database」で止まる。

**原因**  
OneDrive のクラウド同期の影響で、Git がファイルを読むときにタイムアウトすることがある。とくに 02_顧客管理・03_契約・規約 などで出た。

**やった対処**  
1. **一時的に .gitignore に追加** … タイムアウトするフォルダ（例：`02_顧客管理/`、`03_契約・規約/`）を .gitignore に書く。コメントで「OneDrive でタイムアウトするため一時除外。同期が落ち着いたらこの行を削除して git add 可能」と残す。
2. **それ以外だけステージしてコミット** … .gitignore、.cursor、タイムアウトしないフォルダ（04・05 など）だけ `git add` して初回コミットを通した。

**あとで 02・03 も Git に入れたいとき**  
1. OneDrive で 300 フォルダを右クリック → **「常にこのデバイスに保持」** にして同期を完了させる。  
2. .gitignore から該当フォルダの行を削除。  
3. `git add .` してコミット。

## 10. 履歴を確認して、どのような変更をしたかを確認する方法

コミットしたあと、「いつ・何を変更したか」を振り返る手順。

### 方法A：Cursor のソース管理・GitLens から見る

1. **ソース管理パネルを開く** … `Cmd + Shift + G`
2. **コミット履歴を見る**  
   - パネル上部や「…」メニューに **「履歴」「History」「コミット履歴」** などがあればクリック  
   - または左サイドバーの **GitLens** アイコンをクリック → **Commits** や **ファイルの履歴** を開く
3. **一覧からコミットを選ぶ** … 日付・メッセージで「どのコミットか」を特定
4. **そのコミットで何が変わったか見る**  
   - コミットをクリックすると、**変更されたファイル一覧**と**差分**が表示される  
   - ファイル名をクリックすると、そのコミットでの**追加（緑）・削除（赤）**が表示される

### 方法B：ファイル単位で履歴を見る（GitLens）

1. **対象のファイル**をエディタで開く
2. **右クリック** → **「Open File History」**（または GitLens の「ファイルの履歴」）
3. そのファイルの**過去のコミット一覧**が出るので、見たいコミットを選ぶと差分が表示される

### 方法C：ターミナルで確認する

```bash
# コミット一覧（新しい順、1行ずつ）
git log --oneline

# 直近 5 件だけ
git log --oneline -5

# 特定のコミットの変更内容を見る（コミットID は log で出た先頭の英数字）
git show コミットID
```

- `git log --oneline` で出た **先頭の 7 文字程度（コミットID）** を控えておき、`git show コミットID` でそのときの変更内容を表示できる。

### 確認の流れのまとめ

| 知りたいこと | やること |
|--------------|----------|
| 全体で「いつ・どんなメッセージで」コミットしたか | ソース管理 or GitLens のコミット一覧、または `git log --oneline` |
| あるコミットで「どのファイルがどう変わったか」 | そのコミットをクリックして差分表示、または `git show コミットID` |
| あるファイルが「過去にいつどう変わったか」 | ファイルを開いて右クリック → Open File History（GitLens） |

## 11. 過去の履歴から一部を復元して、最新の状態に取り込む方法

「昔のコミットのときのあのファイル（またはあの内容）を、今の作業フォルダに戻したい」ときの手順。

### やりたいことのイメージ

- **例**：3 つ前のコミットのときの `〇〇.md` の内容を、今のフォルダに復活させたい  
- **結果**：指定したコミット時点の `〇〇.md` が、いま開いているワークスペースのファイルとして上書きされる。あとで必要なら「復元した」とメッセージを付けてコミットする。

### 手順（1ファイルを過去の状態に戻す）

#### Step 1：どのコミットに戻すか決める

- 上記「10. 履歴を確認」で、**復元したい時点のコミット**を開く  
- そのコミットの **コミットID**（`git log --oneline` で出る先頭の英数字、例：`a1b2c3d`）を控える  
- Cursor の履歴表示では、コミットをクリックしたときに ID や「Copy Commit ID」などで取得できる場合がある

#### Step 2：そのコミットのファイルを「今のフォルダ」に取り込む

**ターミナルで行う場合**（対象のリポジトリのフォルダで実行）：

```bash
# 例：コミット a1b2c3d のときの「C1_cursor/1b_Cursorマニュアル/260208_〇〇.md」を今の状態に復元
git restore --source=a1b2c3d -- C1_cursor/1b_Cursorマニュアル/260208_〇〇.md
```

- `--source=コミットID` で「どの時点の状態か」を指定  
- 最後の `--` の後ろに **ファイルパス**（リポジトリ内の相対パス）を書く  
- 実行すると、そのファイルだけが指定コミットの内容で**上書き**される（ working tree だけが変わり、まだコミットはしていない状態）

#### Step 3：結果を確認して、必要ならコミットする

1. エディタでそのファイルを開き、**意図どおり復元されているか**確認する  
2. 問題なければ、**通常どおりステージ → コミット**する  
   - 例：「〇〇.md を 2026/02/〇〇 時点の内容に復元」

### 注意点

- **`git restore --source=...` は、指定したファイルを「そのコミットの内容」で上書きするだけ**です。  
  他のファイルや、その後のコミットは消えません。  
- **復元したあと、そのまま編集してからコミット**すれば、「昔の状態をベースにした新しい変更」として履歴に残ります。  
- 複数ファイルをまとめて復元したい場合は、`git restore --source=コミットID -- フォルダ/` のようにフォルダを指定することもできます（そのフォルダ内の全ファイルが指定コミットの状態になる）。

### 別のやり方：中身だけコピーしたい場合

- 「ファイルごと復元」ではなく、**過去のコミットの「あるファイルの内容」をいまの別ファイルにコピーしたい**ような場合は、  
  - 履歴表示でそのコミットのファイルを開いて内容を表示し、**手動でコピー**する  
  - または `git show コミットID:ファイルパス` で中身を表示し、必要な部分だけコピーする  

という方法もあります。
