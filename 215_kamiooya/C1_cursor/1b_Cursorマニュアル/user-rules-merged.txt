# Git コミットの提案と準備

## 前提（ワークスペースに 215_神・大家さん倶楽部 が含まれる場合）
- Git リポジトリは **`~/git-repos`** の 1 つだけ（リモート: project-GE）。
- 編集は OneDrive / Google Drive で行い、`~/git-repos/sync_to_git.sh` で同期してからコミットする運用。

## エージェントの振る舞い

1. **作業が一区切りついたら、コミットを提案する**
   - 「ここでコミットしておくのがおすすめです。」と伝える。
   - **コミットメッセージの例**を 1 つ提示する（日本語で簡潔に）。

2. **提案時に、215 がワークスペースに含まれる場合は同期・ステージをエージェントが実行する**
   - ターミナルで以下を実行し、ソース管理に変更が出ている状態にする：
     `~/git-repos/sync_to_git.sh`
     `cd ~/git-repos && git add -A`
   - 実行が終わったら「同期と git add は完了しました。ソース管理に変更が出ています。」と伝える。
   - 215 が含まれないワークスペースの場合は、同期・ステージの実行は行わず、提案とメッセージ例のみ。

3. **コミット自体はユーザーに任せる**
   - ソース管理画面でメッセージを入力 → ✓ でコミット → Push してもらう。
   - エージェントが `git commit` を代わりに実行しない。

4. **Push もユーザーに任せる（原則）**
   - ユーザーから「Push して」と言われた場合のみ、エージェントが `git push` を実行する。

5. 小さな typo 修正だけなど、**明らかに軽い変更**のときは提案を省略してよい。

## コミットメッセージ例の書き方
- 日本語で短く、何をしたかが分かるようにする。
- 例: 「確定申告マニュアルに〇〇の節を追加」「空室対策メール 260208 を追加」「send_mail.py にエラーハンドリングを追加」

# Gmail MCP チャットでの振る舞い

## 使うツールの優先
Gmail MCP を使うときは、次の4つを優先して使う。
- メールを探す: `search_emails`
- 本文を読む・表示する: `read_email`
- 下書きを作る: `draft_email`
- 送信する: `send_email`
ラベル・フィルタ・一括操作・添付ダウンロードなどは、ユーザーが明示的に依頼したときだけ使う。

## 「この内容で送って」と依頼されたとき
ユーザーが「この内容で送って」「この内容で送信して」「送って」「送信して」「この下書きの内容で送って」などと送信を依頼したときは、**提案された内容（チャット内の下書き・返信文）を使って、新規送信として送る**。
1. チャット内で提案・表示した**宛先・件名・本文**（および必要なら cc / 添付）を確定する。
2. **`send_email`** を呼び、その内容で送信する（`to`, `subject`, `body` 等を指定）。
3. 返信の場合は、取得済みのメールの **threadId** と **inReplyTo**（メッセージID）を `send_email` に渡し、同じスレッドに届くようにする。
- Gmail の「下書き」を ID 指定で送信するツールは存在しないため、「下書きを送信」は行わない。常に**提案内容を send_email で新規送信**する。

# Gmail token 更新時の連携案内

## 適用タイミング
ユーザーが Gmail の token 期限切れ、GMAIL_TOKEN_B64 更新手順、いけともAIニュースの保存が動かない、パートナーとのやりとりで Gmail 認証エラー、のいずれかについて相談しているとき。

## エージェントの振る舞い
手順を案内する**最後に**、次のどちらか（または両方）を **短く 1 〜 2 文で追加**する。
- いけともAIニュースの token 更新を案内した場合: パートナーとのやりとりは 215 の token.json を共有しているため手元で更新すればパートナー用はそのまま。あわせて GitHub の GMAIL_TOKEN_B64 を更新すればいけとも用の定期実行も動くこと。
- パートナーとのやりとりの token 更新を案内した場合: いけともAIニュースの GitHub Actions 用には 215 の token.json を Base64 にしたものを GitHub Secret GMAIL_TOKEN_B64 に登録する必要があること。パートナー用に token を更新したら、DX互助会の `03_outputs/ai_news_save/GitHub Actions用_token更新手順.md` のステップ 2 に従って GMAIL_TOKEN_B64 も更新することをおすすめすること。

参照: 215 は `C1_cursor/1b_Cursorマニュアル/Gmail token 更新時の連携（いけとも・パートナー）.md`、DX互助会は `03_outputs/ai_news_save/GitHub Actions用_token更新手順.md`

# OneDrive 接続が切れたときの振る舞い

ユーザーは OneDrive をメインで利用しており、接続が切れることがよくある。

ファイルの読み書きで **接続エラー**（ETIMEDOUT, connection timed out など）が出た場合：
1. それ以上同じ操作を繰り返さず、いったん止める。
2. すぐにユーザーに「接続が切れています」と伝える。必要なら、内容はここまでで伝えた旨や、接続回復後にやってもらいたいことを1〜2文で補足する。
3. 接続が戻ってから続きを依頼してもらう前提で終える。
- タイムアウト後に同じ読み書きを何度も自動でリトライしない。
- 接続エラーを無視して「代わりにテキストでお渡しします」だけにして、ルール追加などの依頼を忘れない。
