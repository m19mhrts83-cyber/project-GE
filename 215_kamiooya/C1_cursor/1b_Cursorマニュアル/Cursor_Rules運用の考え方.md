# Cursor Rules 運用の考え方（Gems のような機能単位で使う場合）

## 前提：Cursor Rules の仕様

- **ルールの置き場所**: `.cursor/rules/` 直下またはそのサブフォルダ
- **ルールとして読まれるもの**: 拡張子 `.md` または `.mdc` のファイル（サブフォルダ内も対象）
- **注意**: `.cursor/rules/` 内に置いた Markdown は「ルール」として AI に渡されるため、**テンプレート本文や生成用の長文はここに置くと意図せずルールとして適用されやすい**
- **推奨**: ルール定義（何をいつやるか・どこを参照するか）だけを `.cursor/rules/` に置き、**テンプレート・生成用 Markdown はプロジェクトの別フォルダに置いてパスで参照する**

---

## 推奨する構成（2パターン）

### 構成A: ルールは .cursor、中身は既存フォルダで参照（シンプル）

```
.cursor/rules/
├── partner-email-check.mdc      # 既存
├── gmail-token-troubleshooting.mdc
├── git-commit-habit.mdc
├── 機能名1.mdc                  # 新機能：中で「参照」パスを書く
└── 機能名2.mdc

C1_cursor/1b_Cursorマニュアル/   # 既存の「やること」の置き場
├── 機能名1/
│   ├── テンプレート.md          # 生成用テンプレ
│   ├── 出力例.md               # 必要なら
│   └── old/                    # 古いテンプレなど
└── 機能名2/
    ├── テンプレート.md
    └── old/
```

- **よい点**: 今の運用の延長で、ルールは `.cursor/rules/` に集約しつつ、テンプレなどは既存の「Cursorまわり」フォルダで管理できる
- **ルール内の書き方**: 「テンプレートは `C1_cursor/1b_Cursorマニュアル/機能名1/テンプレート.md` を参照し、～に従って生成する」と明示する

---

### 構成B: 機能を一箇所にまとめたい場合（機能フォルダを独立させる）

```
.cursor/rules/
├── partner-email-check.mdc
├── git-commit-habit.mdc
└── 機能名1.mdc                  # 中身で cursor-features/機能名1/ を参照

cursor-features/                 # プロジェクト直下に「機能の資産」を集約
├── 機能名1/
│   ├── テンプレート.md
│   ├── 出力例.md
│   └── old/
│       └── テンプレート_202501.md
└── 機能名2/
    ├── テンプレート.md
    └── old/
```

- **よい点**: 「このプロジェクトで Cursor でやる機能」が `cursor-features/` の下に一覧でき、テンプレ・old を機能ごとにまとめられる
- **ルール内の書き方**: 「テンプレートは `cursor-features/機能名1/テンプレート.md` を使う」「出力は ○○ に保存」などとパスで指定する

## ご質問の「1・2・3」への回答

| 項目 | 回答 |
|------|------|
| **1. 機能ごとにサブフォルダ** | **OK**。ただし「ルール定義」と「テンプレ・生成用Markdown」は分けた方が安全（上記の通り、テンプレは `.cursor/rules/` の外がおすすめ） |
| **2.(a) .cursorrules またはルール定義** | **`.cursor/rules/` に 1機能1ファイルの `.mdc`** を置く形がよい。`.cursorrules` は旧形式で、プロジェクト全体1ファイルになりがちなので、機能ごとに分けるなら `.cursor/rules/機能名.mdc` が向く |
| **2.(b) 生成用 Markdown** | **ルールと同じフォルダではなく、プロジェクト側**（例: `C1_cursor/1b_Cursorマニュアル/機能名/` や `cursor-features/機能名/`）に置き、ルール本文で「このパスを参照して生成する」と書く |
| **2.(c) テンプレートファイル** | 同上。**プロジェクト側の機能フォルダ**に置き、ルールからパスで参照する |
| **3. 古いファイルは old に** | **その運用で問題なし**。`old/` は機能フォルダの下（テンプレを置いている場所）に作ると、機能ごとに履歴が残って扱いやすい |

## ルール定義（.mdc）に書くといいこと（Gems の「指示」に近い）

- **トリガー**: どんな言い方で依頼されたらこの機能を動かすか
- **やること**: 何を実行・生成するか（スクリプト実行／ファイル生成など）
- **参照するファイル**: テンプレート・出力先・設定ファイルの**パス**
- **禁止・注意**: やってはいけないこと、前提条件（認証など）

テンプレの「本文」はルールに貼らず、**「○○のファイルを参照してそれに従え」**と書くだけで、その機能に忠実に動かせます（Gems のテンプレを入れて動かすイメージに近い）。

---

## まとめ

- **ルール（何を・いつ・何を参照するか）** → `.cursor/rules/` に機能ごと `.mdc`
- **テンプレ・生成用 Markdown・old** → プロジェクト側の「機能フォルダ」に配置し、ルールからパスで参照
- **新機能を足すとき** → 新しい `.mdc` を 1 つ追加し、必要なら `C1_cursor/1b_Cursorマニュアル/新機能/` または `cursor-features/新機能/` を作ってテンプレ・old を入れる

この構成にすれば、Gems のように「機能＝テンプレ＋指示」で忠実に動かしつつ、Cursor Rules の仕様にも沿った運用になります。
