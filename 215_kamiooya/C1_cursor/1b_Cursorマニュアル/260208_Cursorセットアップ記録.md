## 0. 拡張機能のインストール（Git 管理・作業効率化）

Git 管理を始める前に、以下の拡張機能を入れておくと、変更履歴の確認やドキュメント編集がしやすくなります。**インストールは `Cmd + Shift + X`（拡張機能）→ 検索で名前 or ID を入力 → インストール。**

### Git 管理に役立つ拡張機能

| 拡張機能名 | 拡張機能 ID | 役割 |
|------------|-------------|------|
| **GitLens** | eamodio.gitlens | 行ごとの変更履歴（Blame）、コミット履歴のグラフ表示 |
| **Rainbow CSV** | mechatroner.rainbow-csv | CSV の列を色分け表示。データ確認に便利 |
| **Better Comments** | aaron-bond.better-comments | コメントを TODO・重要・疑問などで色分け |
| **indent-rainbow** | oderwat.indent-rainbow | インデントを色分け（Python などで見やすい） |

### ドキュメント・ファイルのプレビュー

| 拡張機能名 | 拡張機能 ID | 役割 |
|------------|-------------|------|
| **Markdown All in One** | yzhang.markdown-all-in-one | Markdown のプレビュー・目次・ショートカット |
| **Markdown Preview Enhanced** | shd101wyy.markdown-preview-enhanced | 高品質プレビュー・PDF 出力・mermaid 図 |
| **vscode-pdf** | tomoki1207.pdf | PDF を Cursor 内でプレビュー |
| **Excel Viewer** | GrapeCity.gc-excelviewer | .xlsx を表形式でプレビュー・編集（Wijmo ライセンスダイアログが出る場合は CLOSE で閉じてよい） |
| **Docx Renderer** | AdamRaichu.docx-viewer | .docx を Cursor 内でプレビュー（読み取り専用） |

### 詳細

- 各拡張の使い方・ショートカット・注意点は **「260208_Cursor拡張機能ガイド.md」** にまとめてあります。
- .md を**編集モード**で開くには、プレビューだけになる設定を外す必要がある場合があります。ガイド内の「マークダウン形式とプレビューの切り替え」を参照。

## 2. Git管理のセットアップ

### 2つのリポジトリを作成

#### リポジトリ1: 215_神・大家さん倶楽部（OneDrive上）
- **場所**: `OneDrive-個人用/215_神・大家さん倶楽部/`
- **管理対象フォルダ**:
  - `C1_cursor/` - Cursor関連のコード・設定
  - `C2_ルーティン作業/` - 空室対策メール・管理会社契約
  - `C3_プロンプト/` - AIエージェント用プロンプト
- **除外**: PDF, Excel, 画像, 動画, その他のバイナリファイル
- **注意**: OneDrive上のため `.git` フォルダの同期除外設定を適用済み

#### リポジトリ2: ProgramCode（ローカル）
- **場所**: `~/ProgramCode/`
- **管理対象**: Python スクリプト全般（17ファイル）
- **OneDrive外なので同期の問題なし**

### .gitignore の設定方針
- `/*` で全てを除外してから `!フォルダ名/` で許可する方式（215 の場合）
- テキスト系ファイル（.py, .md, .txt, .html, .sh, .csv）→ 管理対象
- バイナリファイル（.pdf, .xlsx, .jpg, .png, .mov 等）→ 除外

### その他のフォルダ（300・500・DX互助会）の Git 構成
- **215 と同じ「gitdir 構成」**で追加済み。実体の .git は `~/git-repos/` にあり、各クラウドフォルダ内には「gitdir: 〜」と書いた .git ファイルだけを置く。
- 300：`git-repos/300_ai/`、500：`git-repos/500_obsidian/`、DX互助会：`git-repos/dx_kyouyuu/`
- いずれも .gitignore で MD・テキスト・コードを追跡し、バイナリは除外。詳細は「260208_Git管理で学んだこと.md」を参照。

### コミット提案ルール（git-commit-habit）
- 215・300・500・DX互助会・ProgramCode に **`.cursor/rules/git-commit-habit.mdc`** を配置済み。
- AI がファイルを編集したあと、「コミットしておくのがおすすめです」＋メッセージ例を出すが、**コミットはユーザー自身が行う**運用。

## 3. Git操作の手順（自分でやる場合）

### 基本の流れ
**ファイルを編集 → 変更を確認 → ステージング → コミット**

### 手順

| 手順 | やること | ショートカット |
|---|---|---|
| 1 | ファイルを編集する（AI or 手動） | - |
| 2 | ソース管理パネルを開く | `Ctrl + Shift + G` |
| 3 | 変更ファイルをクリックして差分を確認 | - |
| 4 | 「+」ボタンでステージング | - |
| 5 | コミットメッセージを入力（日本語OK） | - |
| 6 | チェックマーク or `Cmd + Enter` でコミット | `Cmd + Enter` |

### コミットメッセージの例
- `メール送信スクリプトにエラーハンドリングを追加`
- `空室対策メール 260208を追加`
- `zaim_to_yayoi.pyの変換ロジックを修正`

### やった方がいいこと
- **こまめにコミットする** - 大きな変更前に保存しておくと失敗しても戻せる
- **メッセージは具体的に** - 後で振り返れるように

### やらない方がいいこと
- credentials やパスワードをコミットしない（.gitignore で除外済み）
- 意味のない変更はコミットしない

### 元に戻したい場合
- **コミット前**: ソース管理パネルでファイル横の「↩（戻す）ボタン」をクリック
- **コミット後**: チャットで「直前のコミットを元に戻して」と指示

## 参考：セットアップ時に AI Agent にした質問

同じようにセットアップを進める人のために、実際に投げた質問の例を分野別にまとめる。このように聞くと、手順や用語の説明が得られやすい。

### Git 運用まわり
- 他のワークスペース（300・500・DX など）にも同じコミット提案ルールを適用してほしい
- コミットメッセージの提案をしてほしい
- エクスプローラーの **U** や **丸** のマークは何を意味するか
- ステージって何か。コミット前に変更内容を確認するにはどうすればいいか
- プッシュ・プル・プルリクエストは今の運用で必要か。Replit など別のクラウドと連携するときはどうか
- フォルダ構成（エクスプローラー）に戻る操作はどうするか
- 300 フォルダだけコミットできずに止まる。どうすればいいか（→ OneDrive タイムアウトの対処）

### ファイル形式・フォルダ構成
- 管理会社への相談を、やり取り履歴＋送信文面で管理したい。MD と TXT のどちらがよいか、運用案を考えてほしい
- 案B（MD 履歴＋TXT 送信下書き）で進めたいので、該当する内容だけ残してほしい
- 古い「ホームプランナーへの相談.txt」は削除してよいか

### エディタ・拡張機能
- .md がプレビューモードでしか開けない。マークダウン形式で編集するにはどうするか（→ 設定で .md を常にプレビューで開く指定を外す）
- Excel をプレビュー表示する拡張機能はあるか。Word も同様にプレビューできる拡張を入れてほしい
- Excel Viewer で「Wijmo ライセンスが無効」と出た。どういう意味か。気にしなくていいか
- Word の中身は AI も確認できるようになるか（→ 拡張でユーザーはプレビュー可能だが、AI は .docx を直接読めない）
- Word がまだプレビューできない。拡張機能のインストール状況を確認してほしい
- Office Viewer と Docx/ODT Viewer のどちらがよいか決め手が欲しい
- 検索で出てきた一覧のなかで、Word プレビューに該当するのはどれか（→ Docx Renderer）
- Docx Renderer を入れたので、拡張機能ガイドの Word まわりを更新してほしい

### Cursor の料金・設定
- プランの上限を超えてしまった。リセットされるのは具体的にいつか
- 次回支払予定日をどこから確認するか
- Auto モードの動きがよくわからない。オートのいいところと、クレジットがあるときは Claude 最新を選んで尽きたら Auto にする運用でよいか
- チャットでモデルを指定していたら課金対象になり追加料金が発生した。運用上の注意としてまとめてほしい

### OneDrive（Mac）
- OneDrive が勝手に接続切れになる。対処方法はないか（Mac に絞って提案してほしい）
- メニューバーに OneDrive アイコンが出てこない。どうすればいいか
- 「常にこのデバイスに保持」は Mac の容量をかなり使う理解で合っているか。容量がないので選べない

### その他
- Notion にページ追加は頼めるか。タスクを追加するときにステータスなども話しながら設定できるか。その場合はプランモードで聞くべきか

## Cursor を個別最適化するための聞き方 Tips

Cursor のよいところは、**自分用に最適化を図れる**こと。以下のような聞き方をすると、やりたいことに合わせた提案や手順が得られやすい。

### やりたいことと制約を一緒に伝える
- **例**：「Word をプレビューで見たい。ただ MD のようにコピペすると背景が崩れるので、そこを避けたい」
- **効果**：「プレビューできる拡張」だけでなく、「コピペしやすい運用」まで含めて提案してもらえる。

### 環境を絞って聞く
- **例**：「OneDrive の対処を、Mac に絞って教えて」「次回支払日はどこで確認するか（Cursor の画面で）」
- **効果**：汎用ではなく、自分の OS・アプリに合った手順だけが出る。

### 選択肢が複数あるときは「決め手」を頼む
- **例**：「Office Viewer と Docx/ODT Viewer のどちらがいいか、決め手が欲しい」「検索で出てきた一覧のなかで、Word プレビューに該当するのはどれか」
- **効果**：比較表や「この条件ならこれ」という結論を出してもらえる。

### うまくいかないときは「状況を確認して」と依頼する
- **例**：「Word がまだプレビューできない。拡張機能のインストール状況を確認して」「300 だけコミットできずに止まる。どうすればいいか」
- **効果**：原因の切り分け（未インストール・設定・OneDrive タイムアウトなど）と対処案を出してもらえる。

### 運用方針を先に言う
- **例**：「案B（MD 履歴＋TXT 送信下書き）で行くので、それに合わせて内容を整理して」「Downloads はルール不要なので削除して」
- **効果**：その方針に沿った変更・整理だけを依頼できる。

### 習慣にしたいことは「適切なタイミングで提案して」と伝える
- **例**：Git 管理を入れたときに「今後は適切なタイミングでコミットするように提案してほしい」と伝えたら、編集後に「コミットしておくのがおすすめです」とメッセージ例を出してくれるようになった。
- **効果**：ルール（git-commit-habit）と合わせて、自分でコミットを忘れにくくなる。ほかの習慣（例：定期的なバックアップの提案など）も、同様に「提案して」と依頼すると取り入れやすい。

### 拡張機能は「役割」で聞き、選び方は続けて聞く
- **例**：「Excel をプレビュー表示する拡張はあるか」→ 候補が出たら「Word も同様にしたい。どれを入れるといいか」
- **効果**：マーケットの候補と、自分環境（既存拡張・容量・ライセンスの心配など）に合う選び方を一緒に決められる。

### あとで使うために「まとめ」「記録」を依頼する
- **例**：「今回の変更のコミットメッセージを提案して」「Git 管理で学んだことを MD にまとめて」「セットアップで何を聞いたか、参考としてまとめて」「今回の聞き方を参考に、こんな聞き方するといいよ、という Tips もまとめて」
- **効果**：同じことをする人や、あとから自分で見返すときの手がかりになる。

### 用語が分からないときはそのまま聞く
- **例**：「ステージって何か」「U や丸のマークは何を意味するか」「プッシュ・プルは今の運用で必要か」
- **効果**：用語の説明と、「今の自分の運用ではどうか」まで一緒に整理してもらえる。
