# Git コミットできなくなった原因調査メモ

**調査日:** 2025年2月28日

---

## 1. 判明した事実

| 項目 | 結果 |
|------|------|
| `.git` の正体 | **ディレクトリではなく、59バイトの「ファイル」** |
| 通常の Git リポジトリ | `.git` は**ディレクトリ**（その中に `HEAD` / `config` / `objects` 等がある） |
| `.git` の読み取り | **タイムアウト**（`Operation timed out` / `error reading .git`） |
| `git status` のエラー | `fatal: error reading .../.git` |

つまり、

- このリポジトリの `.git` は**中身が入ったディレクトリではなく、1個の小さいファイル**になっている。
- そのファイルを開いても**読めない**（OneDrive の同期・ロックの可能性が高い）。

---

## 2. 想定される原因

1. **OneDrive 上の Git との相性**
   - リポジトリが **OneDrive の同期フォルダ内**（`CloudStorage/OneDrive-個人用/215_神・大家さん倶楽部`）にある。
   - OneDrive が `.git` を同期する過程で、
     - `.git` ディレクトリを別の形（ファイルや壊れた状態）に変えてしまった、
     - または `.git` が「オンラインのみ」などで実体がまだ揃っておらず、読み取りがタイムアウトしている、
   といった可能性がある。

2. **worktree の参照切れ（サブパターン）**
   - `.git` が 59 バイトのファイルのとき、中身は多くの場合  
     `gitdir: （実際の .git ディレクトリのパス）` の 1 行。
   - その**参照先**が OneDrive 上の別パスで、そこへのアクセスが遅い／ブロックされていると、  
     「.git を読む」処理がタイムアウトし、`error reading .git` になり得る。

3. **一般的な注意点**
   - Microsoft の Q&A や Tech Community でも「**OneDrive 上に Git リポジトリを置くと不具合・破損が起きやすい**」とされており、  
     **Git リポジトリは OneDrive 外に置くことが推奨**されている。

---

## 3. 対処の方向性（推奨順）

### A. リポジトリを OneDrive の外に移す（いちばん安定）

- 例: ホーム直下や `~/Projects` など、**OneDrive の同期対象外**のフォルダに 215 フォルダを移動（またはクローンし直す）。
- そこで通常の `git` 操作（コミット・プッシュなど）を行う。
- 必要なら、OneDrive 側には「Git 管理しないコピー」や、別のバックアップ方法で同期する。

→ **Git の運用は OneDrive 外、バックアップ・共有は別手段**にすると、同じ問題の再発を防ぎやすいです。

### B. OneDrive を「常にこのデバイスに保持」にしてから様子を見る

- 該当フォルダ（`215_神・大家さん倶楽部`）を右クリックなどで  
  **「常にこのデバイスに保持」**（Always keep on this device）にする。
- 同期が完了し、`.git` がローカルに完全に存在する状態になってから、もう一度 `git status` を実行。
- それでも「.git がファイルのまま」「読めない」なら、**A（OneDrive 外へ移す）**を検討した方が安全。

### C. `.git` が壊れている前提で修復を試す（上級者向け）

- **バックアップを取ったうえで**、現在の `.git`（59バイトのファイル）の**中身を確認**する。
  - ターミナルで:  
    `cat "/Users/matsunomasaharu/Library/CloudStorage/OneDrive-個人用/215_神・大家さん倶楽部/.git"`
  - タイムアウトせずに `gitdir: （パス）` のように出れば、そのパスが「本当の .git の場所」。
- 参照先パスが分かった場合:
  - そのパスが OneDrive 内なら、**A（OneDrive 外へ移す）**を強く推奨。
  - 参照先が無効・存在しない場合は、**このフォルダを新規に `git init` し直し、リモートを再度 `git remote add` して履歴をやり直す**か、別のバックアップから `.git` を復元する必要がある。

---

## 4. まとめ

- **原因:**  
  - `.git` が**ディレクトリではなくファイル**になっている。  
  - そのファイル（またはその参照先）の読み取りが**タイムアウト**しており、**OneDrive の同期フォルダ内にリポジトリがある**ことが主な要因と推測される。
- **おすすめ:**  
  **Git で管理したい「215_神・大家さん倶楽部」は OneDrive の外に置き、そこでコミット・プッシュする。**  
  必要に応じて、OneDrive には Git を使わない別のコピーやバックアップで同期する。

---

## 5. 「フォルダは移せない・履歴はクラウドに残したい」場合の案

- **制約:** 容量の都合で OneDrive 外にフォルダごとコピーできない。ローカルが壊れても履歴を見られるようにクラウドに置いておきたい。

### 案 A: クラウド = GitHub / GitLab にして、作業はローカル 1 か所だけ（推奨）

- **「クラウド」= リモートリポジトリ（GitHub / GitLab / Azure Repos など）** にする。
- 作業用は **OneDrive 外の 1 フォルダだけ**（例: `~/215_神・大家さん倶楽部` や `~/Projects/215`）。そこでは普通に `git commit` / `git push` が動く。
- 履歴はすべてリモートにあるので、**ローカルが壊れても `git clone` し直せば履歴はそのまま見られる**。
- **容量:** リポジトリの実体は「ローカル 1 か所 + リモート」だけ。OneDrive にはこのリポジトリを置かないので、OneDrive の容量を増やさずに済む。必要なら、OneDrive には「Git を使わない別フォルダのバックアップ」だけ置く（履歴はリモートで見る）。

### 案 B: OneDrive の「常にこのデバイスに保持」で .git を完全に同期してから再試行

- 215 フォルダ（および .git が指している参照先があればそのフォルダも）を OneDrive で **「常にこのデバイスに保持」** にする。
- 同期が完全に終わってから、もう一度 `git status` を試す。.git が「オンラインのみ」で読めなかっただけなら、これで直る可能性がある。
- 直らない場合や、また同じ不具合が出る場合は、案 A の運用に切り替えるのが安心。

### 案 C: OneDrive には「成果物だけ」置き、Git の実体はローカル + リモート

- 215 の中身（ドキュメント・成果物）はこれまで通り OneDrive で共有・バックアップする。
- **Git の履歴用**は、案 A と同様に **リモート 1 か所** を正本とし、作業用は OneDrive 外の 1 フォルダのみ。そこから push した履歴を「クラウド（リモート）の履歴」として扱う。
- こうすると「履歴はクラウド（= リモート）に残る」「容量は OneDrive に二重に持たない」の両方を満たせる。
