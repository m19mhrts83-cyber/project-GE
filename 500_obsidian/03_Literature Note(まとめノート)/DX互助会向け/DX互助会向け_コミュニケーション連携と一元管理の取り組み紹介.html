DX互助会向け コミュニケーション連携と一元管理

Gmailだけでなく、複数チャネルを 1 つの運用に統合した実践事例。

  • Gmail API
  • ChatWork API
  • iMessage
  • LINE（個人・グループ）
  • LINE（オープンチャット）

実装手順版は DX互助会向け_コミュニケーション連携_ローカル適用ハンドブック.md を参照。

1. 何を一元管理しているか

flowchart TB subgraph inputs [入力チャネル] Gmail["Gmail API"] ChatWork["ChatWork API"] IMessage["iMessage"] LineTalk["LINE 個人/グループ"] LineOpenChat["LINE オープンチャット"] end subgraph engine [実行層] Cursor["Cursor 依頼 + ルール"] Scripts["Python スクリプト群"] end subgraph outputs [出力先] Yori["パートナー別 5.やり取り.md"] OpenChatLog["815_神大家オプチャ/5.やり取り.md"] end Gmail --> Scripts ChatWork --> Scripts IMessage --> Scripts LineTalk --> Scripts LineOpenChat --> Scripts Cursor --> Scripts Scripts --> Yori Scripts --> OpenChatLog
  • 出力を 5.やり取り.md に統一して、後で読む場所を固定
  • Cursorルールで実行導線を固定し、手順迷子を防止

2. Gmail API / ChatWork API / iMessage

Gmail API

  • 受信取り込み: gmail_to_yoritoori.py
  • 送信: yoritoori_send.py

ChatWork API

  • 取り込み: chatwork_to_yoritoori.py
  • 必要設定: CHATWORK_API_TOKEN, chatwork_room_id

iMessage

  • 取り込み: imessage_to_yoritoori.py
  • macOS依存あり

3. LINE（個人・グループ）

  • 取り込み: chrline_sync_to_yoritoori.py
  • 対象: 1:1 + グループチャット
  • 依存: LINE_UNOFFICIAL_AUTH_DIR と CHRLINE 認証

E2EE の影響で本文が取れないメッセージがあるため、ログは「取れるものを確実に蓄積する」設計。

4. LINE（オープンチャット）

  • バッチ: chrline_open_chat_to_md.py
  • 常駐: chrline_open_chat_realtime_watch.py
  • 手動補完: line_open_chat_thread_clip_to_yoritoori.py

OpenChatスレッドは API 制約があるため、完全自動化できないケースあり。

Beta運用: 自動取得を基本にし、重要スレッドのみ手動補完。

5. ルール設計（再現性の中核）

  • partner-email-check.mdc: 4系統取り込みを実行
  • partner-email-send.mdc: 送信導線を yoritoori_send.py に統一
  • open-chat-archive.mdc: OpenChat取り込み導線を固定
  • 815-openchat-no-reply-proposal.mdc: 返信提案と情報共有を分離

6. ローカル適用の入口

  • 必要ファイル・設定一覧を先に揃える
  • Cursor依頼テンプレで再現確認する
  • 失敗時はチャネル別チェックリストで切り分ける

詳細: ローカル適用ハンドブック

7. まとめ

  • 価値は「手段を増やす」より「増えた手段を同じ運用に乗せる」こと
  • 5チャネル連携 + 1つの記録形式 + ルール設計で再現性を担保
  • OpenChatスレッドは制約前提で Beta 運用する

勉強会では「発表要点 + ハンドブック配布」の2段構成が最も移植しやすい。
