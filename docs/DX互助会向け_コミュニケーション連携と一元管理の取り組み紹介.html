DX互助会向け コミュニケーション連携と一元管理

Gmailだけでなく、複数チャネルを 1 つの運用に統合した実践事例。

  • Gmail API
  • ChatWork API
  • iMessage
  • LINE（個人・グループ）
  • LINE（オープンチャット）

実装手順版は DX互助会向け_コミュニケーション連携_ローカル適用ハンドブック.md を参照。

1. 何を一元管理しているか

2. Gmail API / ChatWork API / iMessage

Gmail API

  • 受信取り込み: gmail_to_yoritoori.py
  • 送信: yoritoori_send.py

ChatWork API

  • 取り込み: chatwork_to_yoritoori.py
  • 必要設定: CHATWORK_API_TOKEN, chatwork_room_id

iMessage

  • 取り込み: imessage_to_yoritoori.py
  • macOS依存あり

3. LINE（個人・グループ）

  • 取り込み: chrline_sync_to_yoritoori.py
  • 対象: 1:1 + グループチャット
  • 依存: LINE_UNOFFICIAL_AUTH_DIR と CHRLINE 認証

E2EE の影響で本文が取れないメッセージがあるため、ログは「取れるものを確実に蓄積する」設計。

4. LINE（オープンチャット）

  • バッチ: line_unofficial_poc/chrline_open_chat_to_md.py
  • 常駐（任意）: line_unofficial_poc/chrline_open_chat_realtime_watch.py

OpenChatスレッドは API 制約があるため、完全自動化できないケースあり。

Beta運用: 自動取得を基本にし、重要スレッドのみ手動補完。

5. オープンチャットの認証まわり（QR 負荷の低減）

※本スライドは暫定です。全体の章立て・表現は、内容を精査したうえで改めて整えます。

改善してきたこと（ねらい）

  • 認証の一本化 … LINE_UNOFFICIAL_AUTH_DIR を、トーク同期（個人・グループ）とオープンチャット取得の両方で共有。同じ chrline_client_utils.build_logged_in_client 経由にすることで、「スクリプトごとに別ログイン → QR が連発」になりにくくした。
  • ローカル既定の保存先 … 未設定時は line_unofficial_poc/.line_auth_local。クラウド同期フォルダ配下に認証を置くと競合しやすく QR が増えやすいため、実行時に警告するようにした。
  • 保存トークンの順試行 … .tokens/ 内の複数ファイルを更新日時の新しい順に試し、有効なものがあれば QR なしで起動する。
  • 再認証後の永続化 … QR で取り直したトークンを .tokens に保存し、次回以降の手間を減らす。
  • 失効の気づき … 例: line_unofficial_poc/launchd/open_chat_healthcheck_runner.sh で軽い疎通チェック＋通知し、気づかないまま連続失敗し続ける時間を減らす。

苦労した点（割り切り）

  • CHRLINE は非公式のため、LINE 側の変更でセッションが切れる。切れたあとの復旧に QR は技術的に省略できない（「回数を減らす」が現実的な上限）。
  • Square（オープンチャット）は取得できる範囲に依存し、スレッド本文が欠けることもある（スライド 4 の Beta 運用とセット）。

任意

  • LINE_CHRLINE_AUTO_OPEN_QR=1 … QR 画像を OS で自動表示（認証回数は減らないが、操作負荷は下げられる）。

6. ルール設計（再現性の中核）

  • partner-email-check.mdc: Gmail / iMessage / ChatWork / LINE / オプチャ取り込みの導線を実行
  • partner-email-send.mdc: 送信導線を yoritoori_send.py に統一
  • open-chat-archive.mdc: OpenChat取り込み導線を固定
  • 815-openchat-no-reply-proposal.mdc: 返信提案と情報共有を分離

7. ローカル適用の入口

  • 必要ファイル・設定一覧を先に揃える
  • Cursor依頼テンプレで再現確認する（参照用プロンプト集あり）
  • 失敗時はチャネル別チェックリストで切り分ける

詳細: ローカル適用ハンドブック / パッケージ参照プロンプト一覧

8. まとめ

  • 価値は「手段を増やす」より「増えた手段を同じ運用に乗せる」こと
  • 5チャネル連携 + 1つの記録形式 + ルール設計で再現性を担保
  • OpenChatスレッドは制約前提で Beta 運用する
  • LINE 系は認証保存先を共有し、QR・手間の再発を抑える（スライド 5）

勉強会では「発表要点 + ハンドブック配布」の2段構成が最も移植しやすい。
