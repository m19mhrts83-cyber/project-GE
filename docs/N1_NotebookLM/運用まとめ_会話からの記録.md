## 3. スライド制作の流れ（依頼 → 結果 → 自分の作業）

文字（表・文案）を軸にしつつ、**ビジュアルは NotebookLM**、**公開 HTML は Cursor ＋ git**、という分担。

### 3.1 自分が Cursor に出す指示（例）

- `@` で材料を指定（既存 `slides_outline`、旧 HTML、下書き MD など）。
- `dx-slides-from-outline.md` にある **定型ブロック**をそのまま貼ると揃いやすい。内容の要旨は次のとおり。
  - `docs/_drafts/slides_outline_<トピック>.md` に表を作る（列: `slide_id | 種別 | 見出し | 本文 | 図案 | 話す一言`）。
  - 同じ返答に **NotebookLM 用パック**（ソース一覧・スタイル指示・最初のチャット用プロンプト・Studio 用追いプロンプト案）。
- MCP を使う場合は、ノート選択と `ask_question` で文案を磨く、と明示する。

### 3.2 Cursor／エージェント側で得られるもの

- `slides_outline` の実ファイル化（依頼すればリポジトリに保存）。
- NotebookLM にコピペする文章一式、必要なら補助 MD。
- MCP 利用時: 登録済みソース前提の回答（**ソースの Web 投入は別**）。
- PNG が `docs/assets/...` に揃った後: HTML の `img` 差し替え等を依頼可能。
- 本フォルダ `N1_NotebookLM` を編集したタスクでは **rsync ミラー**。

### 3.3 自分が必ず／基本やること（ブラウザ・手元）

1. NotebookLM（Web）でノート作成・**ソース追加**・取り込み待ち。
2. チャットに「最初のプロンプト」を貼る。
3. **Studio** で Nano Banana Pro 等を使いスライド／画像を生成。
4. 画像を `s1.png` … のように `slide_id` に合わせて **`docs/assets/<デッキID>/`** に保存。
5. ブラウザでローカル HTML を確認。
6. **`git add` / `commit` / `push`**（公開）。

**MCP だけでは完結しないもの**: ソースのアップロード、Studio 操作、画像のダウンロード／スクショ、git push。

## 4. 認識しやすいポイント（会話での整理）

- **表ができただけでは公開は未完**（PNG と HTML の `src` の対応が必要な場合がある）。
- **NB を使わない**運用も可能（その場合は `dx-slides-from-outline.md` のブロック B を省略し、SVG 等で回す）。
- **ルールファイル**は `.gitignore` の例外により `n1-notebooklm-mirror.mdc` だけリポジトリに含めている（他の `.cursor/rules` は無視のまま）。
